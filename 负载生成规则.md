# 负载生成规则

> Alpha Proposal · Power Profile 模块 目标：根据有限输入，仿真出全年 8760 小时逐时负载曲线，并为后续自建"工厂类型库"提供种子结构。
> 
>

## 需求背景（为什么做这个模块）

**产品定位**：本模块（Power Profile）是 AlphaESS「Alpha Proposal」光储提案工具的一环。Alpha Proposal 面向的 EPC、经销商，终端受益人是工厂主；目标是"约 5 个核心参数、分钟级出初版 ROI 方案"，把出方案周期从传统的几天到一周压缩到当着客户面就能出。

**为什么先要一条负载曲线**：ROI 测算的一切——PV \+ 储能 \+ 柴发微网仿真、削峰、自消纳、停电备电时长、柴油替代收益——都建立在 8760 小时负载曲线之上。负载是整个测算的地基，这也是产品的「load\-first」叙事。

**为什么不能直接让客户上传真实曲线**：缅甸没有智能电表 / NEM12 那类强制区间数据机制，电网也不提供这类型的数据，普通工厂没有负载计量装置且估算非常困难。且当地电表通常只记市电进线，看不到柴发与停电时段的真实负荷，直接用会严重低估真实负载与停电缺口。像 Orkestra、华为 SmartDesign 这类"有真实区间数据才好用"的路线在缅甸走不通。因此本模块用「工厂业态模板 \+ 账单锚定全年总量 \+ 柴油修正 \+ 工作日约束」重建 8760——在拿不到真实区间数据的市场里仍能落地。

其次，通过预设的曲线也能够给出一版相对比较真实的负载信息。客户对于这个工具能生成什么样的内容，以及内容的可靠性都有了一定的把握，因此会更乐于与拿着这个方案的 EPC 推进下一步的工作。

举个例子（之前 Orkestra 也举过类似的例子）：

当把一个范例的东西拿到客户面前时，客户会了解这个事情给他能带来把账算的更清楚的价值，客户会快速地把提资需要的类目找齐，因为他们知道这件事情对他们来说是有正向价值和意义的。



**同步，调研印证的两个客户最关心的关键点**（客户调研 8 份问卷）：

- **ROI 可信度**："财务收益算不清 / 回收期不可信 / 假设难调整"是最集中的不满 → 要求生成的曲线"看起来科学、又加得上账单"，以支撑可信的 ROI。

- **停电 \+ 柴油建模**：进 Top3 痛点，也是华为 / 阳光 / HOMER 等工具配置极复杂或算不准的弱项 → 必须把停电与柴油算进曲线（缅甸刚需、友商弱项）。

> 这两点正是本文档设计取向的由来：两层基载 \+ 账单硬锚（第 2、4 节）服务"可信 ROI"，完整的多台柴发油耗模型（第 6 节）服务"停电 \+ 柴油"。
>

## 用法（先读这一段）

负载分两层：

- **没工作时的负载** → 基载 `B_off`（夜间 / 停工的量）

- **工作时的负载** → 工作基载 `B_work`（开工时的地板）\+ 在其之上升到 `P_peak` 的生产峰

**出曲线的必需输入（缺一不可，只是部分给了默认值）：**

1. **电量** = 月账单电量（电网）\+ 柴油用电量（柴油另需折算，见第 6 节）

2. **负载峰值 ****`P_peak`**（前置步骤复用、可改）

3. **时间设置**：工作时间、工作日 / 周末、节假日、午休 —— 有默认值，但本质上是出曲线的必需项

**可选精修**：淡旺季（生产量比例或 12 月真实电量）、设备清单。

**工作曲线始终由「账单 \+ 实际工作时间段 \+ 形态形状」生成**（第 8 节）。设备清单若提供，只用其**运行时段**细化「实际工作时间段」；**额定功率不直接生成曲线**（额定 ≠ 运行）。

读形态表的方法：取第 9 节某个细类的参数行，代入第 2 节公式，即得逐时表达式；再按第 11 节算法跑全年。

## 设计原则

- **形状 × 标定解耦**：曲线长什么样 与 多大量级 分开处理。

- **两层基载**：`没工作 = B_off`，`工作地板 = B_work`，生产峰在 `B_work` 之上升到 `P_peak`。

- **电量是硬锚**（账单 \+ 柴油，逐月必须分毫不差）；**峰值是软锚**（估算，作峰值目标 \+ 任意小时硬上界）。

- **基载不随淡旺季缩放**：订单掉了，照明 / 空调 / 安防 / 最低产线照常；缩的只有生产峰。

- **额定 ≠ 运行**：设备额定功率不等于实际运行功率，不用来直接拼工作曲线；幅值由账单锚定、形态由类型形状提供。

- **淡旺季只有一种含义**：生产量比例。

## 核心公式

```Plain Text
P(d, h) = P_peak · ( f_base[h] + A_work(m) · f_prod[h] ) · (1 + ε)
```

其中分两层：

```Plain Text
f_base[h] = B_off%                              (h 在工作时间段外)
          = B_work%                             (h 在工作时间段内)

f_prod[h] = 0                                   (时间段外)
          = (1 − B_work%) · g[h] · L[h]         (时间段内)
```

**物理直觉**：夜间 `g=0` 时 `P = P_peak·B_off%`；开工后地板抬到 `P_peak·B_work%`；旺季生产峰处 `g=1、A_work=1`，`P = P_peak`（恰好触峰）。 工作时间段可为**单段或多段 / 循环**（见第 8 节）。两层基载都是 `P_peak` 的固定比例、不随淡旺季缩放；缩放只作用于生产部分 `f_prod`。

## 基载设定（两层、固定值）

**定义**：

- `B_off` = 非工作时段（夜间 / 停工）的量。

- `B_work` = 工作时段的地板（开工但未到生产峰时的水平）。

**B\_off 按夜间档取固定值**（示例，实际参照工厂比例，其他类型工厂套用夜间真停部分）：

**B\_work 按类型取固定值**（见第 9 节每行；通常 40–70%，连续型 = B\_off）。

**硬校验**：每月 `P_peak · (B_off%·非工作小时 + B_work%·工作小时) < E_m`，即基载电量不可能超过当月总电量；违反 → 基载设高或电量填低 → 自动下夹并提示。

---

## 工作层形状 g\[h\]

生产峰形状（在实际工作时间段内做高斯叠加，归一化到峰 = 1）：

```Plain Text
g_raw[h] = Σ_k exp( −(h − μ_k)² / (2 · σ_k²) )     (工作时间段内)，  0 (段外)
g[h]     = g_raw[h] / max_h( g_raw[h] )
```

- **窗口 = 实际工作时间段**（可单段或多段 / 循环，见第 8 节）。

- **尖峰型**（单班制造）：σ 取 1\.5–2\.5，峰间与段边沿 g 低 → 白天起伏明显。

- **近平直型**（连续工艺 / 基站）：σ 取 8–10（很宽），g 在高位窄幅波动 → 全天接近平直。

- **午休因子** `L[h]`：凹陷时段乘 \(1 − δ\)，其余为 1。

- **段边沿**：t\_on / t\_off 处用约 1 h 线性爬坡在 `B_off ↔ B_work` 之间过渡。

> 工作层的真实变化由形态 `g[h]` 提供（而非把设备额定拼成方块），这是不用额定功率直接生成曲线的原因之一。

## 柴油折算——柴油输入若和尖峰负载不匹配，需要弹窗进行一次提醒

柴油模块由「柴发配置（单 / 多台）\+ 停电时间窗 \+ 所选负载类型的基础曲线」一步算出用油量。

### 6\.1 柴发配置

可由系统自动生成，或客户自定义。

**自动生成**（默认弹窗；标准规格 `{100, 300, 500, 1000}` kWA，依尖峰负载 `L = P_peak`）：

- `L ≤ 1000`：取最小的标准规格 `s ≥ L`，超过300KW，小于1000KW后，默认s/2**≥L 。** 例：L=400 → 2×300；L=1000 → 3×500。

- `L > 1000`：**⌈L / 1000⌉ 台 × 1000 kW**。 例：L=1\.5 MW → 2×1000；L=2\.5 MW → 3×1000。

**客户自定义**：

- 可任意配置机型组合，如 `150 kWA × 1`、`指定功率 × N 台`、或大小搭配（最多两种不同型号）。

- 修改配置后，尖峰负载小于柴发功率，则弹提醒，否则以柴发配置功率为准：以柴发总容量 `P_gen_total = （Σ 机组）*0.8` ＜P\_peak。

- 允许客户对于每台柴发的P25%、50%、75%、100%的效率进行填写

### 6\.2 多台柴发调度（80% 装载阈值）

每台在线机组尽量保持在 **≤ 80% 额定装载**（效率区间，留裕度、避免低载湿堆积与过载）。据此决定每小时开几台、哪几台。

**\(a\) 大小搭配型**（大机 `P_big` \+ 小机 `P_small`）：

- `P_small*0.8＞P_load(h) `：只开小机代载。

- `P_small*0.8＜P_load(h) ≤ 0.8 · P_big`：只开大机代载。

- `P_load(h) > 0.8 · P_big`：大机 \+ 小机并机分担。

**\(b\) 同型号并机型**（n 台 × s）：

- 在线台数 `k(h) = min( n, ⌈ P_load(h) / (0.8 · s) ⌉ )`，即每台 ≤ 80% 装载所需的最少台数。

- 在线机组**等分负载**：每台 `load_i = P_load(h) / k(h)`。

- 随负载升降增减台数。

- 若负载的功率超过了柴发可以带载的量，自动按照100%Pn的负载耗油量进行求解

### 6\.3 逐时用油量（按在线机组的部分负载曲线求和，负载曲线按照选择的负载类型和峰值功率去求解，不循环带入该模块生成的曲线）

**负载曲线 ****`P_load(h)`**** 仅由「所选负载类型 \+ 峰值功率」求解**（即 `P_load(h) = P_peak · u_type[h]` 的代表日基础曲线），**不回带本模块生成 / 标定后的曲线**——这是为了避免循环引用。

```Plain Text
Fuel(h) [L/h] = Σ_{在线机组 i}  b(r_i) · load_i(h)
```

- `r_i = load_i(h) / s_i` 为该机装载率；`b(r_i)` 是单位油耗（L/kWh），由该机型油耗表**按装载率插值**得到。

- **不含空载截距**：`load_i = 0`（离线或零出力）→ 该机不耗油。

- 默认油耗表（无厂家数据时，线性插值；越轻载单位油耗越高）： 

### 6\.4 汇总（仅在停电时段累加）

```Plain Text
总用油量      V        = Σ_{h ∈ 停电窗}  Fuel(h)
柴油供电量    E_diesel = Σ_{h ∈ 停电窗}  P_served(h)
```

- 不存在停电缺口

### 6\.5 与电量锚的衔接

- 设施总电量 = 电网电量（账单）\+ 柴油供电量 `E_diesel`。

- 账单对应电网供电的小时：标定时令 `∫_{电网在线小时} P_load = 账单 kWh`；停电小时的曲线电量即 `E_diesel`，自然导出（见第 11 节第 4 步）。

### 6\.6 简化回退（无停电窗 / 无配置时）

- 客户直接给柴油**用油量\(升\)** → `E_diesel = V × η`，`η` 默认 **3\.5 kWh/升**（轻载约 3\.0，区间 3\.0–3\.8）。

- 客户直接给柴油**用电量\(kWh\)** → 直接用。

- 此回退不做逐时油耗、调度与停电缺口，仅把柴油电量并入总电量。

## 工作曲线如何生成（额定 ≠ 运行）

工作曲线**不**由「设备额定功率 × 运行时段」直接加和得到。原因：设备额定功率不等于实际运行功率，两者之间存在差距；硬加额定值会同时**高估幅值**、又**过于方块化**（缺少真实变化）。

工作曲线由三个要素生成：

1. **账单数据**（电网 \+ 柴油）→ 锁定幅值 / 电量（通过 A\_work 校准，见第 11 节）。

2. **实际的工作时间段** → 决定工作层在哪些时段"在线"（单段或多段 / 循环，见下）。

3. **基载的形态形状 ****`g[h]`** → 提供工作时段内的真实变化形态。

即仍走第 2 节核心公式：工作时段 = `B_work% + (1 − B_work%)·A_work·g[h]·L[h]`，时段外 = `B_off%`。

**设备清单的作用（重新定义）**：

- 若客户提供设备清单，其**运行时段**用于细化「实际的工作时间段」（要素 2）；

- 其**额定功率不参与幅值计算**；

- 如确需用额定功率给不同时段做相对加权，应乘一个运行系数 `η_util ≈ 0.4–0.7`（运行功率 = 额定 × η\_util），但幅值的最终锚仍是账单。

**多段 / 循环工作时间**：「实际的工作时间段」支持最多三段或循环（如 06:00–10:00 \+ 14:00–18:00）。工作层（B\_work \+ 生产峰）在任一运行段内出现，所有段之外回落到 B\_off；形态 `g[h]` 在各运行段内应用。第 9 节表中的单段窗口是各类型的**默认值**，用户 / 设备排班可将其细化为多段。

> 若客户能提供一段**真实实测曲线**，则直接采用、优先级最高（覆盖形态形状）—— 这是最准的来源。

## ~~淡旺季处理（"真账单"模式，只保留生产量比例）第一步先不做~~

**优先级：真账单 \> 相对值。**

1. **12 个月真实月度电量**：客户逐月给出真实电量 → 直接据此区分各月，逐月校准 `A_work(m)` 使每月电量分毫不差。最准。

2. **只给生产量比例**：客户给月份 \+ 相对值（生产量 / 订单比例）→ 按比例缩放生产部分。

```Plain Text
A_work(m) = w_m / w_ref         （w = 该月相对生产量；典型 / 旺季月 w_ref = 1）
```

- 只缩生产部分 `f_prod`，**基载（B\_off、B\_work）不动**。

- 单张账单时的月电量分配：`E_m = E_base_m + (w_m / w_ref) · E_prod_ref`，其中 `E_base_m` 为当月两层基载电量。

## 生成算法（执行顺序）

1. **解析电量** → 逐月 `E_m = E_grid_m + E_diesel_m`（账单按电价反推电网电量；柴油按第 6 节折算）。取 `P_peak`。

2. **定形态与工作时间段**： 

    - 查第 9 节得 `B_off% / B_work% / 运行窗口（默认实际工作时间段）/ 午休 / 产量峰`；

    - 若客户给设备清单 → 用其运行时段把「实际工作时间段」细化为多段 / 循环（**不改幅值、不改形态形状**）；

    - 若客户给真实实测曲线 → 直接采用，跳过形态。

3. **建日模板**：在实际工作时间段（单 / 多段）内算 `g[h]`（归一化峰 = 1）与午休 `L[h]`；段边沿在 `B_off ↔ B_work` 间爬坡。

4. **锁电量（硬锚，校准生产系数 A\_work，基载不动）**： 

    - 12 月真实电量 → 逐月 `A_work(m) = (E_m − E_base_m) / E_prod_raw_m`； `E_base_m = P_peak·(B_off%·非工作小时 + B_work%·工作小时)`， `E_prod_raw_m = P_peak·(1 − B_work%)·Σ_工作 g·L`。

    - 仅相对值 → `A_work(m) ∝ w_m`，整体缩放使年总 = 年电量。

    - **校验** `E_m > E_base_m`，否则基载过高 → 提示。

5. **拼 8760**：`P = P_peak·(f_base + A_work·f_prod)`，按日历分配工作日 / 周末 / 节假日（周末关停 → 全天 B\_off；半天班 → 缩短时间段）。

6. **加纹理 ε**（逐日幅度对数正态 ±8% \+ 小时级 AR\(1\) φ≈0\.6），压在 P\_peak 以下，再做一次逐月电量归一化以保账单。

7. **和解峰值**：旺季生产峰处 `g=1、A_work=1` → 自然 `max(P) ≈ P_peak`；偏差 \> 10% 则微调 `B_work` 或产量峰 σ。校验日负载率是否落在合理区间。

8. **输出** 8760 逐时曲线 \+ 校验报告（逐月 kWh = 账单 ✓、年负载率、峰值落点）。

9. **输出工作日和非工作日典型曲线，给客户进行查看（不可修改）**

## 校验与提示

- `LF = ΣE / (P_peak·8760)` 必须 ∈ \(0,1\)。`≥ 1` → 电量配不上这个峰值（峰值偏低或电量偏高，注意电量已含柴油）；`< 0.15` → 峰值多半虚高。

- 每月 `E_m > E_base_m`，否则两层基载设高。

- 反算逐月 kWh 必须 = 账单 \+ 柴油（硬约束）。

- `max(P)` 与 `P_peak` 偏差 \> 10% → 提示并给出和解后的峰值。

- 日负载率落在形态区间外 → 可能选错形态或输入异常，提示。

## 作为类型库种子与后续校准

第 9 节的形态表即"工厂类型库"的初始 schema：每个细类带 `B_off% / B_work%`、运行窗口、午休、产量峰，与类型图一一对应。 后续每接一个新客户，用其真实间隔数据反推参数（拟合 B\_work、μ/σ、产量峰），逐步把默认值替换成数据驱动的真实 archetype，并对参数表做版本化迭代。短期用规则形态起步，长期沉淀为可校准的库，两者无缝衔接、不需重写。
